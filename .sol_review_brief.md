## What the change does (verify, don't trust this summary)

This is a Rust workspace for a fast, local-first macOS mail client. A prior slice
added an **account-identity gate**: before syncing, it hashes the connected
Google account's immutable OIDC `sub` and, if a *different* account now owns the
encrypted mailbox store, clears it. This slice (3d-3b) adds an **in-transaction
identity fence** so that a sync cycle that started under account A can never
persist writes after the store's recorded identity has moved to account B.

Mechanism:
- `run_identity_gate` (adapters/oauth-sync-macos/src/lib.rs) now returns the
  committed identity hash as a `fence: IdentityHash`. `gated_sync` threads it into
  `SyncCoordinator::sync_recent`.
- `MailboxStore::reconcile_recent_envelopes` and `cache_message_if_present`
  (crates/application/src/mailbox.rs) gained a `fence: &IdentityHash` param.
- `SyncCoordinator` (crates/application/src/sync.rs) threads the fence to both
  store writes and maps `MailboxStoreError::IdentityChanged` ‚Üí
  `SyncFailureSource::IdentityFenced` via `store_failure_source`.
- The SQLCipher store (adapters/store-sqlcipher-macos/src/lib.rs) opens each
  writer txn `BEGIN IMMEDIATE` and calls `assert_identity_unchanged(&txn, account,
  fence)` just before commit on BOTH `reconcile` and `cache_if_present`.

## Focus your review on

1. **Correctness of the fence guarantee.** Can any code path persist an envelope
   or body write when the recorded `account_identity` no longer equals the fence?
   Consider: the ordering of `assert_identity_unchanged` relative to the mutations
   and the commit; the `BEGIN IMMEDIATE` lock timing; whether `Match` returning
   `fresh` (vs the stored hash) can ever differ from the recorded row.
2. **Fail-closed edges in `assert_identity_unchanged`.** Missing row, wrong
   `algo_version`, wrong blob length, NULL columns ‚Äî does every anomalous case
   fail closed (abort the write), never silently pass or re-baseline?
3. **The gate‚Üísync handoff.** Is there a TOCTOU window between the gate committing
   the identity and the first fenced write that the fence does NOT cover? (Note: a
   per-slot whole-cycle permit is explicitly deferred to 3d-3c; the fence is meant
   to hold even without it. Judge whether that claim is true.)
4. **Error-category leakage.** Does `IdentityFenced` correctly distinguish from
   `Store(_)`? Any place a real storage fault is miscategorized as a fence abort
   or vice-versa?
5. **Test integrity.** Do the new tests actually exercise the abort+rollback (not
   just the error return)? Any test that would pass even if the fence were removed?

Ignore pure style. This is security-adjacent (cross-account mailbox isolation);
weigh correctness and fail-closed behavior heavily.
